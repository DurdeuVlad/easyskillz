# v2: Instruction File Management

Deferred from v1. Design decisions made, waiting for user demand signal.

## The Problem

Tools like Claude Code, Codex, Gemini CLI each read from their own root instruction
file (CLAUDE.md, AGENTS.md, GEMINI.md). These files:

- Clutter the repo root
- Get duplicated across tools when content is identical
- Conflict between developers with different tool setups
- Don't work consistently in subfolders (Windsurf/AGENTS.md known issue)

## Two User Profiles

**Personal** — doesn't care about instruction files. Just wants skills centralized.
Instruction management is noise for this user.

**Professional** — already has CLAUDE.md, AGENTS.md, complex structure, subfolders.
Has real pain. Needs a proper solution, not a half-built one.

## Decided Design

### Two config files
- `easyskillz.json` — committed, shared team config (tool list, instruction mode)
- `easyskillz.local.json` — gitignored, machine-only (link strategy, local prefs)

### Instruction management is opt-in, asked once on first sync
```
Manage your instruction files (CLAUDE.md, AGENTS.md, etc.)? [y/N]
```

If yes:
```
Centralize shared instructions into one shared.md for all tools? [Y/n]
```

Stored in easyskillz.json:
```json
{
  "manageInstructions": true,
  "sharedInstructions": true
}
```

Never asked again after first sync.

### Four end states

| manageInstructions | sharedInstructions | Behavior |
|---|---|---|
| false | — | easyskillz never touches instruction files |
| true | true | shared.md → synced to all root files |
| true | false | per-tool files → each root file independently |
| true | true + per-tool override | shared.md base, claude.md extends/replaces for that tool |

### Detection logic on first sync (if manageInstructions = true)

| What's on disk | What happens |
|---|---|
| Nothing | Creates shared.md with starter template |
| One file (e.g. CLAUDE.md) | Moves content → shared.md, replaces root with content + pointer |
| Multiple identical | Merges to shared.md, replaces all roots with content + pointer |
| Multiple different | Moves each to per-tool file, empty shared.md, each root gets content + pointer |
| Already managed | Re-syncs content from source. Idempotent. |

### Root file format (content + pointer, not just stub)
```md
<!-- Managed by easyskillz. Source: .easyskillz/instructions/shared.md -->
<!-- Run `easyskillz sync` to update. Do not edit this file directly.  -->

[actual content inline so tools can read without knowing about easyskillz]
```

### Source files location
```
.easyskillz/instructions/
  shared.md        ← always applied to all tools
  claude.md        ← optional per-tool override
  gemini.md        ← optional per-tool override
```

### Auto-prompt to sync
Lives in the meta-skill and pointer comments. No silent behavior.
```
Before pushing, run: `easyskillz sync`
When instruction files change, run: `easyskillz sync`
```

## Open Questions for v2

- Subfolder support — some tools read from cwd, not root. Sync copies to subfolders?
- Watch mode — `easyskillz watch` auto-syncs on file save
- `easyskillz open instructions` — opens source file in $EDITOR
- What happens when a user has nested repos or monorepos?

## Why Deferred

Half-built instruction management is worse than none. The skills centralization
is the clear universal value. Ship that clean, let users request instruction
management when they actually hit the pain.
