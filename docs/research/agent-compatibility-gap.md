# easyskillz Agent Compatibility Research

Date: 2026-05-08

## Executive Summary

The concern is valid. easyskillz currently treats "skill support" as mostly a path problem: put the same `SKILL.md` folder under each tool's configured skills directory, then append one generic managed block to the tool instruction file. Current agent ecosystems do not work that way anymore.

The biggest mismatch is Codex. easyskillz wires Codex to `.codex/skills`, but current OpenAI Codex docs say repository skills are discovered from `.agents/skills`, walking from the current directory up to the repo root. That means easyskillz can successfully create a symlink that Codex simply will not use.

The second major mismatch is Cursor. easyskillz wires Cursor to `.cursor/skills`, but official Cursor docs describe `.cursor/rules/*.mdc` and `AGENTS.md` as the supported instruction mechanisms. I found no current official Cursor evidence that `.cursor/skills` is a native discovery path.

The third major mismatch is instruction-file centralization. The current `docsStrategy: "unified"` turns root `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md` into tiny stubs pointing at `.easyskillz/docs/INSTRUCTION.md`. Several agents load only the actual instruction file content they discover. They are not guaranteed to follow a "see this other file" pointer, and some explicitly do not support include chaining. In this repo, the centralized file has also absorbed those stubs, so the canonical instruction source is partly self-referential.

## Local Code Findings

### Current Model

The registry is the whole integration model. Each tool has one `skillsDir`, one `instructionFile`, and optional extra wiring:

- Claude: `.claude/skills`, `CLAUDE.md`
- Codex: `.codex/skills`, `AGENTS.md`
- Cursor: `.cursor/skills`, `AGENTS.md`
- Windsurf: `.windsurf/skills`, `AGENTS.md`, plus `.windsurf/workflows`
- Copilot: `.github/skills`, `.github/copilot-instructions.md`
- Gemini: `.gemini/skills`, `GEMINI.md`

Relevant code:

- `src/registry.js`
- `src/wirer.js`
- `src/docs/syncFolder.js`
- `src/docs/centralizeFiles.js`
- `src/docs/scanInstructionFiles.js`
- `src/init/execute.js`

### Wiring Behavior

`src/wirer.js` wires each source skill from `.easyskillz/skills/<name>` into the registered tool's `skillsDir`. It uses symlinks when available, with stub fallback. For Windsurf workflows only, it links `SKILL.md` as a flat `.md` file.

The generic stub fallback is weak for real agent consumption:

```md
# skill-name

> Managed by easyskillz...
> Read the actual skill from: `.easyskillz/skills/skill-name/SKILL.md`
```

That is not a skill adapter. It is a pointer. Agents that index descriptions/frontmatter at startup may see no useful metadata, no real body, and no bundled resources.

### Skill Metadata Behavior

`ensureValidSkill()` only checks whether `SKILL.md` starts with `---`. If not, it injects:

```yaml
---
name: <skillName>
description: autogenerated so ai agents can see the skill
---
```

This is too shallow. It does not validate:

- required fields per host
- host-specific optional fields such as Claude `disable-model-invocation`, Claude/Windsurf/Copilot `allowed-tools`, or provenance metadata
- description quality, which several agents use as the activation trigger
- lowercase/hyphen naming beyond `easyskillz add`
- whether resources are still discoverable when a tool receives a stub instead of the real directory

### Instruction Docs Behavior

`centralizeUnified()` merges all scanned instruction files in a folder into `.easyskillz/docs/<folder>/INSTRUCTION.md`, then replaces each original with either a symlink or a stub pointer.

This repo currently has:

- `AGENTS.md`: stub only
- `CLAUDE.md`: stub only
- `GEMINI.md`: stub only
- `.easyskillz/docs/INSTRUCTION.md`: merged content containing copies of those stubs plus repeated managed blocks

That is broken in practice. For agents, the discovered file needs to contain the instructions, not merely point somewhere else unless the host explicitly implements include/import processing.

### Detector Behavior

The detectors mostly check existence of the registry path or instruction file. If the path itself is wrong, detection can still look healthy while runtime loading fails. Example: Codex can be "registered" because `.codex` exists, while skills are wired to `.codex/skills`, which current Codex docs do not list as a repository skill path.

## External Findings By Agent

### OpenAI Codex

Codex supports `AGENTS.md` for custom project instructions. The broader AGENTS.md convention is plain Markdown, has no required fields, and supports nested files where the closest file wins.

Current Codex skills docs say repository skills are loaded from `.agents/skills` directories along the path from CWD to repo root, plus user/admin/system locations. They also state duplicate skill names are not merged.

Gap:

- easyskillz uses `.codex/skills`, likely stale.
- easyskillz does not emit `.agents/skills`, the current cross-agent path shared by Codex, Copilot, and Windsurf.
- easyskillz replaces `AGENTS.md` with a pointer when docs are unified; Codex needs real loaded instruction text.

### Claude Code

Claude Code's native skill path is `.claude/skills/<skill>/SKILL.md` for project skills and `~/.claude/skills` for personal skills. Skills use progressive disclosure: descriptions are available up front; full `SKILL.md` loads when invoked. Claude Code supports additional frontmatter such as `description`, `when_to_use`, `disable-model-invocation`, `user-invocable`, `allowed-tools`, `argument-hint`, and arguments. It also supports nested `.claude/skills` discovery.

Gap:

- `.claude/skills` wiring is basically right.
- The generic metadata injector is too weak for high-quality Claude behavior.
- Stub fallback is risky because Claude expects a real skill directory with `SKILL.md` plus supporting files.
- Unified docs can weaken `CLAUDE.md` by replacing actual content with a pointer.

### Cursor

Cursor's current official rules model is not `SKILL.md` folders. It supports:

- Project Rules in `.cursor/rules`, written as `.mdc` files with metadata and content.
- User Rules in Cursor settings.
- `AGENTS.md` as a simple Markdown alternative for straightforward agent instructions.
- Legacy `.cursorrules`, deprecated.

Gap:

- `.cursor/skills` is probably a dead or unsupported target.
- easyskillz does not generate `.cursor/rules/*.mdc`.
- A shared `SKILL.md` cannot be treated as a Cursor Project Rule without compiling metadata to MDC fields such as description/globs/alwaysApply.

### Windsurf

Windsurf supports several distinct customization types:

- Rules: `.windsurf/rules/*.md` with `trigger` frontmatter (`always_on`, `model_decision`, `glob`, `manual`).
- `AGENTS.md`: discovered anywhere in the workspace, processed by the same Rules engine; root is always-on, subdirectories become auto-glob rules.
- Workflows: `.windsurf/workflows/*.md`, manual-only slash commands.
- Skills: `.windsurf/skills/<skill>/SKILL.md`, global `~/.codeium/windsurf/skills`, and cross-agent `.agents/skills` / `~/.agents/skills`.

Gap:

- `.windsurf/skills` wiring is valid.
- `.windsurf/workflows` wiring is not semantically valid for every skill. Workflows are manual prompt templates, not progressive-disclosure skill folders.
- easyskillz does not generate `.windsurf/rules/*.md` for behavioral guidance.
- It misses `.agents/skills`, which Windsurf explicitly supports for cross-agent compatibility.

### GitHub Copilot

Copilot supports multiple instruction and skill systems depending on host:

- `.github/copilot-instructions.md` for repository-wide instructions.
- `.github/instructions/**/*.instructions.md` for path-specific instructions with frontmatter.
- `AGENTS.md`, `CLAUDE.md`, or `GEMINI.md` as agent instructions in Copilot cloud agent.
- Agent skills in `.github/skills`, `.claude/skills`, or `.agents/skills`, plus personal `~/.copilot/skills` or `~/.agents/skills`.
- Copilot CLI custom agents under `.github/agents` and user agents under `~/.copilot/agents`.

Gap:

- `.github/skills` is valid for Copilot skills.
- easyskillz does not generate `.github/instructions/*.instructions.md` for scoped/path-specific guidance.
- It does not use `.agents/skills`, again missing the common cross-agent path.
- It does not track Copilot skill provenance metadata used by `gh skill update`.

### Gemini CLI

Gemini CLI uses hierarchical context files, defaulting to `GEMINI.md`, configurable through `context.fileName`. It loads global, ancestor/project, and subdirectory context files, concatenates them with origin separators, and supports `/memory reload` and `/memory show`. The official Gemini CLI reference also has `gemini skills` commands and a `/skills reload` command, but easyskillz currently assumes `.gemini/skills` without validating current discovery paths from the skill docs.

Gap:

- `GEMINI.md` is the right default instruction file, but replacing it with a pointer is fragile.
- easyskillz does not configure Gemini to read `AGENTS.md` via `.gemini/settings.json`, even though the AGENTS.md standard documents that option.
- `.gemini/skills` may not be enough; current Gemini CLI docs also expose skill management commands and should be treated as a moving integration target.

## Hypotheses

1. Codex skills are currently broken because easyskillz writes to `.codex/skills` instead of `.agents/skills`.

2. Cursor skills are currently fake support. Cursor needs `.cursor/rules/*.mdc` and/or `AGENTS.md`; `.cursor/skills` is not supported by current official docs.

3. Unified instruction centralization is unsafe. It assumes agents follow stub pointers, but agents generally ingest discovered instruction files as text. If the content is only "See: path", the agent may not load the actual instructions.

4. One central `SKILL.md` can work only as source IR, not as the emitted artifact for every tool. easyskillz needs per-host adapters.

5. `.agents/skills` should become a first-class output target. It is now the clearest cross-agent skill path across Codex, Copilot, and Windsurf.

6. The current metadata injector creates syntactically valid but semantically poor skills. Activation depends heavily on description quality, and "autogenerated so ai agents can see the skill" is too weak to trigger reliably.

7. Stubs should be treated as compatibility degradation, not equivalent wiring. A host that supports bundled files needs the real directory, not a pointer file.

## Recommended Direction

### Replace Path Registry With Capability Registry

Each tool should declare capabilities, not just paths:

- `instructionFiles`
- `skillDirs`
- `ruleDirs`
- `workflowDirs`
- `agentDirs`
- `supportsAgentsMd`
- `supportsAgentSkillsSpec`
- `supportsSymlinkedDirs`
- `supportsStubPointers`
- `metadataSchema`

### Add Emitters

Use `.easyskillz/skills/<name>/SKILL.md` as the source format, then emit host-specific artifacts:

- `agent-skills`: `.agents/skills/<name>/SKILL.md`
- `claude-skill`: `.claude/skills/<name>/SKILL.md`
- `windsurf-skill`: `.windsurf/skills/<name>/SKILL.md`
- `copilot-skill`: `.github/skills/<name>/SKILL.md`
- `cursor-rule`: `.cursor/rules/<name>.mdc`
- `windsurf-rule`: `.windsurf/rules/<name>.md`
- `windsurf-workflow`: only for explicit manual workflows, not every skill
- `copilot-instruction`: `.github/instructions/<name>.instructions.md` when scoped rules exist

### Stop Replacing Instruction Files With Pointer Stubs

For `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md`, generate real content in place or create real symlinks only. If symlink creation fails, copy generated content instead of writing a pointer.

### Make `AGENTS.md` The Shared Instruction Target

Generate root and nested `AGENTS.md` files as plain Markdown. For Gemini, optionally configure `.gemini/settings.json` to use `AGENTS.md` as the context filename if the user chooses that mode.

### Validate Against Host Schemas

Add a `easyskillz doctor` or `easyskillz project status` command that checks:

- target paths match current docs
- emitted files contain real instruction bodies
- skill frontmatter has useful `name` and `description`
- no centralized instruction doc has recursive stub content
- resources are available in emitted skill directories
